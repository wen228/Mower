# 产品计划书

---

## 1. 项目背景与目标

### 背景

电动工具对BLDC和IoT的需求涌现，主要源于性能升级和智能化转型两大核心驱动力，它们正在共同重塑这个传统行业。
1，BLDC天然效率高、寿命长、噪音低。近年来，**核心组件成本下降推动普及**，随着半导体技术的发展，正在成为趋势。
2，IoT核心价值，**从“使用”到“管理”**。1，远程监控：通过App实时监控工具状态，进行远程诊断和预防性维护。2，创造生意：催生了设备租赁、能源管理等新全新服务模式。

### 目标

**短时间内**交付一副**可演示**的产品原型，体现：

- **工程**：能落地、会用 AI、展示技术和方法  
- **产品**：会定范围、分阶段、取舍  
- **业务**：不堆技术玩具，对准真实需求  

### 用户，问题，解决方案？
| 维度 | 说明 |
|------|------|
| **用户是谁** | **操作者**；**资产管理员**；**公司后台** |
| **用户问题** | **操作者**：作业场景复杂，转速难控+不安全；**管理员**：怕昂贵资产丢失与损坏；**公司**：要可学习的作业数据 |
| **解决方案** | 作业智能调档；异常安全停机；资产状态实时监控；事后本地/云端回看数据 |

## 3. 分阶段功能规划
### 分阶段规划
| 阶段 | 焦点 | 挂钩的用户问题 |
|------|------|----------------|
| **一、主控电机逻辑** | 档位、软启、负载、故障闩锁 | 操作不安全 → 先**可控、可保护** |
| **二、主控相关外设** | 人机、本地记录、自检与状态 | 状态不透明 → **可感知、可复核** |
| **三、联网** | 云快照、上传、无线大屏等 | 要数据 / 要看见 → **先调通**，逻辑保持简单 |

## 4. 技术路线（产品理解）

选型只回答一件事：怎样更快验证第 1 章的用户价值。

| 选型 | 产品理由 |
|------|----------|
| **CoreS3** | LVGL界面，MVC架构明了，不怕AI乱改。二次开发快，适合验证操作与状态呈现 |
| **Unit Roller 等 BLDC 模组** | 支持 Grove I2C 快速搭、亦可走向更高电压工业接口；**API 丰富，闭环/PID 已调且可调** → 团队精力放在工具行为与保护逻辑，而不是重写 FOC |
| **ESP 生态** | Wi‑Fi / MQTT / ESP-NOW 等路径成熟 → **第三阶段联网能尽快调通**，先服务「数据到后台 / 状态到大屏」 |
| **Atom + 外接屏** | 状态一眼可读，服务管理/演示场景；可延展后续智能展示 |
| **开源模块** | 缩短到「稳定可演示」的时间 |

## 5. 为什么有差距（取舍）

早期草稿偏「功能愿望清单」，落地时以**用户问题是否被推进**和**能否按时可演示**为准，因此出现有意差距。

| 方向 | 草稿/设想 | 实际策略 | 取舍理由 |
|------|-----------|----------|----------|
| 输入与交互 | 编码器等人手输入 | 以触控主路径为主，编码器类放弃 | 不解决主路径用户问题，还引入双总线稳定性风险 |
| 电机能力叙事 | 强调算法/自适应功率等 | 应用层档位、限流、负载与闩锁 | **不做 PID 表演**；安全可控比算法秀更贴工具业务 |
| 联网深度 | 全字段看板、长连接、复杂编排 | 关键字段快照 + 文件上传 + 无线屏，**先通** | 公司要的是数据闭环起点；深逻辑可后置 |
| 资产与健康 | 健康分 0–100、寿命模型 | SOC/用电量等可演示的使用度量 | 完整寿命模型超出短周期原型；先建立**可记录、可上传** |
| 状态可视 | 仅本机小屏 | 增加外接大屏同步 | 强化「管理员/演示场景看得见」，面试与业务观感一致 |
| 结构与硬件 | 物料表中的多种选配 | 主路径 CoreS3 + 电机模组 + 必要外设 | **不打板、不扩复杂结构**，保证敏捷交付 |

**结论：** 范围压在「控制逻辑 + 可调通的物联网」，对齐 Onekey 智能化业务的延伸方式。
