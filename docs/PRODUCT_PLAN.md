# Smart Lawn Mower · 产品计划书（轻量）

> 短周期可演示原型：智能控制逻辑 + 物联网。  
> 功能细节见 [README](../README.md) · 演示素材见 [DEMO](./DEMO.md)。

---

## 1. 项目背景与目标

### 背景

TTI 旗下电动/园艺工具正从「能用」走向「可控、可监测、可联网」。本项目不从零设计电机芯片或整机导航，而是在开源硬件上快速验证：**应用层智能控制 + IoT 可见性**，贴近智慧工具生态的产品逻辑。

### 目标

在**尽量短的时间内**交付一副可演示的产品原型，同时体现：

- **工程能力**：能落地、能联调、能稳定演示  
- **产品能力**：会定范围、会分阶段、会取舍  
- **业务贴合**：用 IoT 与状态数据服务工具智能化叙事，而非堆技术玩具  

### 做什么 · 怎么用 · 解决什么

| 维度 | 说明 |
|------|------|
| **用户是谁** | **现场操作者**；**资产管理者**；**公司后台**（收集作业与运行数据） |
| **场景** | 作业中调档与启停；异常时安全停机；现场或大屏看状态；事后本地/云端回看使用数据 |
| **用户问题** | **操作者**：工具「只会转」时难控、不安全；**管理员**：担心昂贵资产丢失与损坏；**公司**：需要可学习的作业/运行数据支撑迭代 |
| **我们做什么** | 应用层**智能控制逻辑**（可控、可保护、状态可解释）+ **物联网**（本地记录、云端快照、无线看板）。实现细节见仓库 README，本文不展开功能清单 |
| **怎么用** | 触控/档位操作 → 运行中监测与故障闩锁提示 → REC 本地记录 → 云端机队字段或大屏同步查看 |
| **对 TTI 的意义** | 贴合 **Onekey** 等智能化业务线方向：在**已有、领先的智慧工具生态**上延伸「安全控制 + 资产状态 + 数据闭环」，而不是另起一套与业务无关的极客 demo |

## 2. 范围界定（做 / 不做）

为保证最短路径出原型，范围刻意收窄。

| 做 | 不做 |
|----|------|
| **智能控制逻辑**（应用层：档位策略、保护与闩锁、人机状态） | **PID / BLDC 底层算法**（使用模组现成闭环与可调参数） |
| **物联网**（本地数据 + 云/无线可见，**调通优先**） | **打板、复杂机械结构** |
| **开源硬件二次开发** + 小代码敏捷迭代 | 大而全平台、重业务中台、长连接复杂逻辑 |

原则：能证明「工具更智能、更可管、数据可沉淀」即可；不做电机厂商该做的底层，也不做整机厂结构件。

## 3. 要解决什么 + 分阶段功能规划

### 要解决什么（产品层）

把「旋转的电机」变成「**可被正确使用、异常可拦截、状态可看见、行为可记录**的智能工具节点」——从而同时回应操作安全、资产风险与公司数据需求。

### 分阶段规划

| 阶段 | 焦点 | 主要回应的用户问题 |
|------|------|--------------------|
| **一、主控电机逻辑** | 档位与启停、软启、负载感知、故障闩锁 | 操作者「只会转、不安全」→ 先建立**可控、可保护** |
| **二、主控相关外设** | 人机界面、本地记录、自检与状态呈现 | 管理员/操作者「状态不透明」→ **可感知、可复核** |
| **三、联网** | 云快照、文件上传、无线大屏等 | 公司「要数据」、现场「要看得见」→ **任何联网以调通优先**，不做复杂业务编排 |

说明：实际开发是快速迭代，阶段可有重叠；与早期草稿不完全一致之处见第 5 章。功能条目以仓库交付为准，此处只定**阶段意图**。

## 4. 技术路线（产品理解）

技术是手段，目的是更快验证第 1 章的用户价值。

| 选型 | 为什么有助于解题 / 出原型 |
|------|---------------------------|
| **CoreS3 + 官方 LVGL UserDemo** | 自带界面框架与 Page/MVC 式结构，**二次开发成本低**，适合快速验证「操作者怎么点、状态怎么呈现」 |
| **Unit Roller 等 BLDC 模组** | 支持 Grove I2C 快速搭、亦可走向更高电压工业接口；**API 丰富，闭环/PID 已调且可调** → 团队精力放在工具行为与保护逻辑，而不是重写 FOC |
| **ESP 生态** | Wi‑Fi / MQTT / ESP-NOW 等路径成熟 → **第三阶段联网能尽快调通**，先服务「数据到后台 / 状态到大屏」 |
| **Atom + 外接显示** | 运行状态**一眼可读**，提升演示与展陈观感；架构上也为后续更智能的展示（含 AI/LLM 类能力）留口子 |
| **开源模块拼接 + 工程供电** | 缩短「能跑」到「能稳定演示」的时间；避免把周期耗在打板与结构上 |

一句话：**站在应用层与 IoT 边界做产品**，把底层运动控制交给成熟模组，把差异化放在控制策略、可观测性与数据闭环。

## 5. 为什么有差距（取舍）

早期草稿偏「功能愿望清单」，落地时以**用户问题是否被推进**和**能否按时可演示**为准，因此出现有意差距。

| 方向 | 草稿/设想 | 实际策略 | 取舍理由 |
|------|-----------|----------|----------|
| 输入与交互 | 编码器等人手输入 | 以触控主路径为主，编码器类放弃 | 不解决主路径用户问题，还引入双总线稳定性风险 |
| 电机能力叙事 | 强调算法/自适应功率等 | 应用层档位、限流、负载与闩锁 | **不做 PID 表演**；安全可控比算法秀更贴工具业务 |
| 联网深度 | 全字段看板、长连接、复杂编排 | 关键字段快照 + 文件上传 + 无线屏，**先通** | 公司要的是数据闭环起点；深逻辑可后置 |
| 资产与健康 | 健康分 0–100、寿命模型 | SOC/用电量等可演示的使用度量 | 完整寿命模型超出短周期原型；先建立**可记录、可上传** |
| 状态可视 | 仅本机小屏 | 增加外接大屏同步 | 强化「管理员/演示场景看得见」，面试与业务观感一致 |
| 结构与硬件 | 物料表中的多种选配 | 主路径 CoreS3 + 电机模组 + 必要外设 | **不打板、不扩复杂结构**，保证敏捷交付 |

### 复盘一句

差距不是「没做完作业」，而是把范围压回：

> **智能控制逻辑 + 可调通的物联网**，服务操作安全、资产可管、数据可学——对齐 Onekey/智能化业务线的延伸方式。

后续若继续，应优先加深「数据对后台是否真有用」「保护策略是否更贴真实工况」，而不是先加更多技术模块。
