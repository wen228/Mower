# FA: USB 供电不稳 → 黑屏 / 串口 Disconnect

**现象时间线：** 引入 WiFi + EzData2 MQTT 后出现  
**结论：** 优先为 **USB 5V 供电余量不足（含 brownout 复位）**，非 LVGL「卡死」、非 MQTT topic 问题  
**验证：** DinBase **12V** 给底座/动力侧供电后，长时间运行无黑屏、无 disconnect  

---

## 1. 现象

| 表现 | 说明 |
|------|------|
| 瞬间黑屏 | 整屏灭，非界面冻住 |
| 串口 disconnect | USB 设备消失，几乎无异常栈日志 |
| 日志位置 | 常在 **尚未出现 `[EZ2]`** 或刚要上云前后；业务 `run= rpm=` 仍在打时突然断 |
| 与 WiFi | 引入云上传后才明显 |

**卡死 vs 复位：**

- **卡死：** 串口通常还在，可能继续刷 `run=...`  
- **复位/掉电：** 串口 disconnect + 瞬间黑屏 ← 本例  

---

## 2. 判断链路

```
USB 5V 同时：屏 + CoreS3 + (可选)BLDC 取电 + WiFi 连网/发包尖峰
        ↓
5V 电压塌陷 / 电流触顶
        ↓
ESP32 brownout 或硬复位
        ↓
黑屏 + USB 重枚举（主机侧见 disconnect）
```

引入 WiFi 后尖峰更频；**未到完整 MQTT 会话** 也可能在 `WiFi.begin`/射频起来时就掉。

**已排除/低优先：**

| 项 | 原因 |
|----|------|
| MQTT 订阅 / topic 未建 | 云侧已能 RX `code:200`，与黑屏无关 |
| 单纯 LVGL 刷新 bug | 一般不断 USB |
| 看门狗为主因 | 上传可阻塞数秒～十余秒有 WDT **理论**风险；但「无 `[EZ2]` 就断」更符合掉电 |

---

## 3. 数量级（粗算）

| 项目 | 量级 |
|------|------|
| WiFi 发射尖峰（模组侧） | ~200–400 mA @3.3V，~**0.5–1.5 W** |
| 屏+主控空闲（USB 5V） | 常 ~150–300 mA |
| USB 口舒适供电 | 多按 **≤500 mA** 设计，线材虚则更差 |
| BLDC 经 USB/5V 取电 | 可达 **安级**，远大于 WiFi |
| 调试版一次云同步阻塞 | ~**5–15 s**（连 WiFi + pub + 双 wait GET） |
| Task WDT（ESP32 常见） | 常 ~**5 s** 量级（可配），≠业务 15 s |

结论：WiFi 是 **瓦级尖峰**；USB 同时供电机时是 **安级风险**。WiFi 常是压垮临界供电的最后一下。

---

## 4. 对策

| 措施 | 状态 |
|------|------|
| DinBase **12V** 供电（动力/底座与 USB 弱供分离） | **已验证有效** |
| 避免单 USB 同时重载电机 + WiFi 尖峰 | 建议作为 demo 默认 |
| 软件（可选后续） | WiFi 保持连接；缩短 debug wait/GET；启动打印 `esp_reset_reason()` |

---

## 5. 再发时快速对照

1. 黑屏是否伴随 **disconnect**？→ 先查供电/复位  
2. 最后日志有无 `[EZ2]`？→ 有则兼查阻塞/WDT；无则更偏掉电  
3. 仅 USB 是否复现、12V 是否消失？→ 坐实电源  

---

## 6. 一句话

> **CoreS3 经 USB 弱供 + 电机负载 + WiFi 尖峰 → 掉电/brownout → 黑屏与串口断开；DinBase 12V 供电后故障消除。**
