# SD 日志：关文件、掉电、长时间 REC 的风险说明

> 对应实现：`src/sd/SdLogger`（手动 REC/STOP，约 200ms 一行）  
> 用途：面试/自用笔记，非代码规范。

---

## 1. 正常时文件怎么关？

`STOP` / `toggle` 关录时大致是：

1. `file_.flush()` — 尽量把缓存刷进卡  
2. `file_.close()` — 关文件、更新 FAT 目录项  
3. （可选）`SdUnmount` 引用计数减掉  

**只有走完 close，目录里才有一份“完整、可被电脑稳定打开”的文件。**  
`REC` 只是 `open` + 写表头；中间靠 `print` 持续追加。

---

## 2. 卡死 / 掉电，文件会丢吗？

**不一定整文件都没，但经常是“半残”或“看不见/打不开”。**

| 情况 | 常见结果 |
|--|--|
| 已 `close` 过 | 一般安全，数据在 |
| 正在写、只 `print` 过、很久没 `flush` | 最后几行～几十行可能还在 RAM 缓冲里 → **丢** |
| 写了很多但从未 `close` 就死机 | 扇区里可能有数据，但 **FAT 目录未正确更新** → 电脑上像 0 字节、打不开、或需修盘 |
| 我们每 **10 行 flush 一次** | 减轻“全丢”，**不能**保证最后不足 10 行的数据；也**不能**代替 `close` |

所以：

- **不是**“卡死就一定整文件 0 字节”  
- **是**“未正常 close 时，文件完整性没保证；尾部最容易丢”

这和串口不同：串口发出去主机已经收到；写 SD 是 **先缓存 → 卡 → 再更新文件系统元数据**，中间断电最脆的是元数据。

---

## 3. “像串口每帧记，STOP 才落盘”？——不完全是

当前实现更接近：

- **REC 之后**：大约每 **200ms** 就 `printf` 一行到打开的 `File`  
- **每 10 行** 额外 `flush` 一次（尽量落盘）  
- **STOP**：再 `flush` + `close`（正式收尾）

所以 **不是** 一直只存在内存、STOP 才一次性写入；  
而是 **边录边往文件系统写**，STOP 是 **收尾关文件**。

若一直不 STOP、但机器一直活着：文件会越写越大，目录里往往也能看到在变（取决于 flush/驱动），但 **规范做法仍是 STOP 再拔卡**。

---

## 4. 一直 REC 不 STOP，会怎样炸？

| 风险 | 说明 |
|--|--|
| **卡空间满** | `print`/`open` 失败；若没仔细处理，可能停更或状态异常 |
| **文件过大** | 200ms 一行 ≈ 5 行/秒；一行约几十字节 → 约 **几十 KB/分钟**。几小时通常还能忍；几天连续可能很大但不立刻 exception |
| **堆/文件句柄** | 只开 **一个** `File`，一般不因“录太久”泄漏句柄；风险更在 **卡满/ IO 错** |
| **看门狗 / 卡顿** | 写 SD 偶发变慢，和 LCD 抢 SPI 时 UI 顿一下（少见 hard fault） |
| **flash 磨损** | 消费级 TF，狂写几年才是寿命问题；面试 demo 可忽略 |
| **没有“最大行数”保护** | 当前实现 **没有** 自动封顶停录；理论上可一直写到失败 |

**很少**会因为“录太久”直接 C++ exception（Arduino 上也不常用异常）；更常见是 **返回 false / 写失败 / 卡死在 SPI**。

粗算：5 Hz × ~40 B ≈ **200 B/s ≈ 12 KB/min ≈ 0.7 MB/小时**。  
8GB 卡理论上能录很久，**先撞满或异常，而不是先“行数上限”**。

---

## 5. 其它边际条件 / 风险

1. **未 STOP 就拔卡** — 和死机类似，可能损坏尾部或目录。  
2. **REC 中离开 AppSD** — 设计上仍可在 `main` 里 `poll` 继续写；挂载靠引用计数，别误以为离开就停录。  
3. **REC 中再插拔卡** — 句柄失效，后续写可能挂；应 STOP 再插。  
4. **和列表扫描同时** — 同卡上 list + 写，一般能扛，极端时 IO 错乱（概率低）。  
5. **LCD ∥ SD 共 SPI** — 狂刷屏 + 狂写卡时更容易卡顿（CoreS3 已知限制）。  
6. **多任务/中断** — 现在单线程 loop，相对简单；若以后 RTOS 多任务写同一 `File` 要加锁。  
7. **时间戳用 `millis()`** — 溢出约 49 天；断电重来；不是 RTC 墙钟。  
8. **断电前最后 <10 行** — 可能还在缓冲（直到下次 flush/close）。  
9. **无卡点 REC** — 应失败并提示（`no card` / `mount fail`），不应当成功。  
10. **文件名扫最大编号** — 删中间文件会跳号；编号到 99 有回绕逻辑需心里有数。

---

## 6. 一句话

> 日志是 **打开文件后周期 append + 定期 flush，STOP 时 close 保证文件系统一致**；  
> 死机/拔卡 **可能丢尾部或文件不完整**，不是事务数据库；  
> 长时间 REC 主要风险是 **卡满和 IO 失败**，不是无限行数的语言 exception。

若以后要更稳：更勤 `flush`、写完关键事件立刻 flush、或 **环形缓冲 + 定期 close 换新文件**（按大小/时间切分）。当前实现是面试够用的工程折中。
